# blueheron.gallery — PHP site (the live storefront)

This `web/` folder is the whole public website. It's plain PHP — no React, no
build step, no framework. Upload it to any PHP host and it runs.

## The split

- **PHP (this folder) = the live site.** Home, gallery, field guide, the order
  page, Stripe checkout, and the admin order screen. This is what you edit and
  maintain.
- **Python (`../server`, `../core`) = your local studio tool.** It protects
  photos (invisible watermark + crop/title) and publishes the finished images
  plus `data/photos.json` into this folder. You run it on your own machine; you
  never deploy it.

So the everyday flow is: run the Python tool locally to prep photos → it drops
protected images + `data/photos.json` into `web/` → upload `web/` to your host.

## One nav, everywhere

The navigation lives in exactly one file: [`partials/nav.php`](partials/nav.php).
Every page does `include partials/nav.php`, so the menu is written once and
can't drift. Change a link there and it changes site-wide. (Same for
`partials/head.php` and `partials/footer.php`.)

## Run it locally

Any PHP 8.1+ works:

```bash
cd web
cp config.local.php.example config.local.php   # then edit it
php -S localhost:8808          # open http://localhost:8808
```

## Files

```
web/

  facts.php          quick facts
  order.php          print picker → Stripe checkout
  order-success.php  post-payment thank-you
  guide/*.php        field guide (generated by ../scripts/build_guide.py)
  api/
    checkout.php     creates a Stripe Checkout Session (prices server-side)
    webhook.php      Stripe webhook → records the order
  admin/index.php    password login + order fulfillment
  partials/          nav.php · head.php · footer.php   ← shared, edit once
  lib/               data.php · db.php · auth.php
  assets/            styles.css · order.css · guide.css
  data/              photos.json · catalog.json   (photos.json comes from the studio tool)
  images/web, images/thumb   (protected images from the studio tool)
  storage/orders.db  (created by the webhook; not deployed)
  vendor/            stripe/stripe-php (composer)
  config.php         reads config.local.php (your secrets, git-ignored)
```

## Products & pricing

Everything the store sells lives in [`data/catalog.json`](data/catalog.json):
each format is a label, a blurb, and a `size → price` map. Add canvas sizes,
a mug, a phone case — it's just JSON, and both the order page and the price
math read from it. (Prodigi can fulfil all of those; see the top-level README.)

## Turning Stripe on

1. `config.local.php` → set `stripe_secret` to your `sk_test_…` key.
2. `composer install` if `vendor/` isn't present.
3. Point a Stripe webhook at `https://yoursite/api/webhook.php` for the
   `checkout.session.completed` event, and put its `whsec_…` secret in
   `config.local.php` as `stripe_webhook_secret`. For local testing:
   `stripe listen --forward-to localhost:8808/api/webhook.php`.
4. Test with card `4242 4242 4242 4242`. Orders appear in `/admin`.

With no key set, the order page degrades gracefully to an email order — the
site still works, it just doesn't take card payments yet.

## Deploy

Upload `web/` to your PHP host (set the document root to it). Make sure
`storage/` is writable. Keep `config.local.php` off any public repo. Run
`composer install --no-dev` on the host if `vendor/` isn't uploaded.
